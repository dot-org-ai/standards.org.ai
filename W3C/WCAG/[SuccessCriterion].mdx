## Intent

### Purpose

This success criterion ensures that {criterion.name.toLowerCase()} so that:

- Users with disabilities can perceive and understand content
- Content is operable by all users regardless of input method
- Information is presented in ways users can comprehend
- Content works reliably with current and future technologies

### Who Benefits

This criterion primarily helps:

- **Screen Reader Users**: People who are blind or have low vision
- **Keyboard-Only Users**: People with motor disabilities
- **Voice Control Users**: People using speech recognition software
- **Cognitive Disabilities**: People who benefit from clear structure
- **Mobile Users**: People accessing content on touch devices
- **Elderly Users**: People with age-related impairments

### Business Benefits

Meeting this criterion provides:

- Expanded market reach to users with disabilities
- Improved SEO and search engine compatibility
- Better usability for all users
- Reduced legal and compliance risks
- Enhanced brand reputation
- Future-proof content architecture

## Testing Procedures

<TestingProcedure criterion={criterion.code} />

### Manual Testing

1. **Visual Inspection**
   - Review the page/component layout
   - Identify elements covered by this criterion
   - Check for obvious violations

2. **Keyboard Testing**
   ```
   - Navigate using Tab key only
   - Activate controls with Enter/Space
   - Verify all interactive elements are reachable
   - Check that focus order is logical
   ```

3. **Screen Reader Testing**
   - Test with NVDA (Windows) or VoiceOver (Mac)
   - Navigate through all content
   - Verify information is announced correctly
   - Check that relationships are clear

4. **Visual Verification**
   - Test at 200% zoom level
   - Check in high contrast mode
   - Verify with color blindness simulators
   - Test on different screen sizes

## Common Challenges and Solutions

### Challenge 1: Dynamic Content

**Problem**: Content that changes without page reload may not be announced.

**Solution**:
```html
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
>
  <!-- Dynamic content updates announced to screen readers -->
</div>
```

### Challenge 2: Complex Widgets

**Problem**: Custom widgets may not have proper accessibility semantics.

**Solution**:
```jsx
const AccessibleTabPanel = () => {
  const [activeTab, setActiveTab] = useState(0)

  return (
    <div>
      <div role="tablist" aria-label="Content sections">
        {tabs.map((tab, index) => (
          <button
            key={index}
            role="tab"
            aria-selected={activeTab === index}
            aria-controls={`panel-${index}`}
            id={`tab-${index}`}
            onClick={() => setActiveTab(index)}
          >
            {tab.label}
          </button>
        ))}
      </div>

      {tabs.map((tab, index) => (
        <div
          key={index}
          role="tabpanel"
          id={`panel-${index}`}
          aria-labelledby={`tab-${index}`}
          hidden={activeTab !== index}
        >
          {tab.content}
        </div>
      ))}
    </div>
  )
}
```

### Challenge 3: Form Validation

**Problem**: Error messages not associated with form fields.

**Solution**:
```html
<div class="form-field">
  <label for="email">Email Address</label>
  <input
    type="email"
    id="email"
    aria-describedby="email-error email-help"
    aria-invalid="true"
  />
  <span id="email-help">We'll never share your email.</span>
  <span id="email-error" role="alert">
    Please enter a valid email address.
  </span>
</div>
```
